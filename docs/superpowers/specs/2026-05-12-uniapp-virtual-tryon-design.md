# Uni-app 工具型试衣试色小程序设计文档

- 日期：2026-05-12
- 主题：`uni-app` 小程序，包含 `在线试衣` 与 `在线口红试色` 两大模块
- 项目定位：工具型体验小程序，不承载商品交易
- 目标版本：可商用初版，首版以演示为主，但结构需支持后续平滑升级上线

## 1. 设计结论

本项目采用 `uni-app 小程序端 + 业务后端 + AI 能力适配层 + 运营后台` 的四层架构，面向匿名用户提供两类能力：

1. 在线试衣
2. 在线口红试色

首版支持两种使用模式：

1. 用户上传本人照片体验
2. 用户选择模板模特体验

首版素材来源采用双通道：

1. 内置官方素材库
2. 后台运营上传素材

用户无需登录，按匿名会话使用。系统重点追求结果真实性，接受较长生成时延，因此所有 AI 生成能力统一走异步任务流。AI 接入采用“双 AI 能力中台”方案：试衣与试口红分别接入更擅长的第三方服务，业务后端统一封装，对小程序端暴露稳定接口，便于后续切换供应商。

## 2. 范围定义

### 2.1 首版包含

1. 首页双入口：在线试衣、口红试色
2. 本人上传模式
3. 模板模特模式
4. 衣服素材库浏览与选择
5. 口红色号库浏览与选择
6. 图片上传、任务创建、任务轮询、结果展示、结果保存
7. 后台管理衣服素材、口红色号、模板模特、上下架、排序
8. AI 服务适配层，支持未来替换供应商

### 2.2 首版不包含

1. 商品交易、购物车、下单、支付
2. 用户账号体系、收藏、长期历史记录
3. 社区内容、评论、分享广场
4. 多人协同编辑后台
5. 推荐算法和个性化画像

## 3. 业务目标

### 3.1 用户目标

用户无需注册即可快速获得较真实的试衣或试口红效果，并能保存结果图用于个人参考。

### 3.2 业务目标

1. 用较低的使用门槛展示 AI 试衣与试色能力
2. 用统一业务后端隔离前端与第三方 AI 差异
3. 建立可运营的素材管理机制
4. 为后续正式上线预留隐私、风控、审核和供应商切换空间

### 3.3 成功标准

1. 用户可在不登录的情况下完整跑通试衣流程
2. 用户可在不登录的情况下完整跑通试口红流程
3. 本人上传与模板模特两种模式都可稳定使用
4. 生成流程对前端是统一任务制，不依赖同步等待第三方 AI
5. 更换 AI 供应商时，不需要改小程序端接口

## 4. 整体架构

### 4.1 架构分层

#### A. Uni-app 小程序端

负责：

1. 页面展示与交互
2. 上传图片
3. 选择模式与素材
4. 提交任务
5. 轮询任务状态
6. 展示生成结果并允许保存

约束：

1. 不直接调用第三方 AI
2. 不承载复杂图像处理逻辑
3. 只调用业务后端提供的统一接口

#### B. 业务后端

负责：

1. 匿名会话管理
2. 上传文件管理
3. 素材读取接口
4. 任务创建与状态管理
5. 异步队列调度
6. AI 适配层统一封装
7. 结果图回写与对外返回
8. 基础限流、错误处理、日志记录

#### C. AI 能力适配层

拆为两个内部适配器：

1. `virtual-fitting adapter`
2. `lipstick-tryon adapter`

职责：

1. 屏蔽不同供应商的入参与返回差异
2. 将业务任务转换为供应商请求
3. 统一错误码和失败原因
4. 为后续替换供应商提供单一改造点

#### D. 运营后台

负责：

1. 衣服素材管理
2. 口红色号管理
3. 模板模特管理
4. 素材上下架和排序
5. 任务状态查看与失败排查

### 4.2 关键架构原则

1. 前端只面向业务后端，不直接接第三方服务
2. AI 处理统一异步化，避免小程序同步阻塞
3. 试衣与试色共用任务中心，但保留独立适配器
4. 匿名可用，但上传数据和生成结果需自动过期清理
5. 素材系统先满足运营维护，再考虑个性化扩展

## 5. 页面结构与用户流程

### 5.1 页面结构

首版建议包含以下页面：

1. `首页`
2. `试衣页`
3. `试口红页`
4. `结果页`
5. `生成中状态页` 或结果页内嵌加载态
6. `后台管理端` 页面集合

### 5.2 首页

展示内容：

1. 在线试衣入口
2. 口红试色入口
3. 热门衣服推荐
4. 热门色号推荐
5. 示例模特快捷入口

目标：

1. 用户第一屏即可理解两个核心能力
2. 尽量缩短到首次生成任务的路径

### 5.3 试衣页

用户流程：

1. 选择模式：上传本人照片 / 选择模板模特
2. 如果选择本人照片，则上传正面、光线正常、背景尽量简单的全身或半身照
3. 浏览并选择衣服素材
4. 提交试衣任务
5. 进入生成中状态
6. 跳转结果页查看效果

页面能力：

1. 模式切换
2. 衣服分类筛选
3. 素材卡片预览
4. 照片上传引导
5. 提交前校验

### 5.4 试口红页

用户流程：

1. 选择模式：上传自拍 / 选择模板模特
2. 如果选择本人照片，则上传正脸、无遮挡、嘴唇清晰的自拍照
3. 浏览并选择品牌、系列、色号
4. 提交试色任务
5. 进入生成中状态
6. 跳转结果页查看效果

页面能力：

1. 模式切换
2. 品牌与系列筛选
3. 色号列表展示
4. 自拍上传引导
5. 提交前校验

### 5.5 结果页

统一承接试衣与试口红结果。

功能包括：

1. 原图 / 效果图切换
2. 前后对比滑杆
3. 显示当前素材信息
4. 重新选择素材后再次生成
5. 保存结果图到本地

说明：

1. 首版不做长期历史记录
2. 会话期内可以保留最近一次或少量最近结果用于回看

## 6. 后端接口与任务流

### 6.1 核心接口分类

#### 素材接口

1. 获取衣服列表
2. 获取口红列表
3. 获取模板模特列表
4. 获取单个素材详情

#### 上传接口

1. 接收用户上传原图
2. 存储图片并返回 `imageId`

#### 任务接口

1. 创建试衣任务
2. 创建试口红任务
3. 查询任务状态
4. 获取任务结果

### 6.2 任务状态模型

统一任务状态：

1. `pending`
2. `running`
3. `success`
4. `failed`

状态说明：

1. `pending` 表示已受理但尚未执行
2. `running` 表示已进入 AI 生成阶段
3. `success` 表示结果已产出
4. `failed` 表示生成失败，允许用户重试

### 6.3 标准任务流

1. 用户上传本人照片，后端存储原图并返回 `imageId`
2. 用户选择素材，前端提交 `imageId`、`assetId`、模式、任务类型
3. 后端创建 `taskId`，写入任务表，初始状态为 `pending`
4. 后端将任务投递到异步队列
5. Worker 取出任务，调用对应 AI 适配器
6. AI 返回结果图后，后端回写状态为 `success`
7. 若调用失败或超时，状态写为 `failed`
8. 前端轮询 `taskId` 获取最新状态
9. 任务成功后，前端读取结果图并进入结果展示

### 6.4 为什么采用任务制

1. 高真实度图像生成耗时波动较大
2. 小程序端不适合长时间同步等待
3. 任务层便于重试、告警、限流和审计
4. 替换供应商时可保持前端接口稳定

## 7. AI 接入策略

### 7.1 方案结论

采用“双 AI 能力中台”方案：

1. 试衣能力使用更擅长服饰换装的第三方服务
2. 试口红能力使用更擅长面部美妆试色的第三方服务

### 7.2 适配层要求

每个适配器对外统一暴露以下语义能力：

1. 参数校验
2. 请求构造
3. 发起生成
4. 状态跟踪
5. 结果标准化
6. 错误标准化

### 7.3 替换策略

未来若切换供应商，只允许在适配层改动：

1. 供应商请求参数映射
2. 签名与鉴权逻辑
3. 错误码映射
4. 结果图回传解析

小程序端和业务层任务接口不应因此变化。

## 9. 匿名会话与隐私处理

### 9.1 匿名会话

首版不要求微信登录。后端为每个小程序访问会话分配匿名 `session_id`，用于关联：

1. 上传图片
2. 生成任务
3. 会话期内短期结果回看
4. 简单限流

### 9.2 隐私原则

1. 上传前明确告知图片仅用于试衣或试色生成
2. 不做用户画像与长期身份绑定
3. 不长期保存原图
4. 原图与结果图都设置自动清理策略
5. 后台仅面向管理员，不默认开放用户原图浏览能力

### 9.3 清理策略

建议默认设置：

1. 原图在 `1-7 天` 内自动删除
2. 结果图在 `1-7 天` 内自动删除，具体周期按存储成本和运营需求定
3. 任务表可保留元数据更久，用于统计与排障，但不保留长期可访问的用户图片

## 10. 异常处理与风控

### 10.1 输入校验

试衣场景：

1. 非正面照提示重新上传
2. 光线过暗提示重新上传
3. 人体主体不清晰提示重新上传

试口红场景：

1. 非正脸提示重新上传
2. 嘴唇区域不清晰提示重新上传
3. 面部遮挡明显提示重新上传

### 10.2 任务异常

1. 供应商超时：任务标记 `failed`，提示稍后重试
2. 供应商返回错误：任务标记 `failed`，记录标准化失败原因
3. 结果图无效：按失败处理，不向前端返回损坏资源
4. 素材已下架：提交时以后端校验为准并提示重新选择

### 10.3 风控与限流

匿名场景下首版至少要有：

1. 基于 `session_id` 的频率限制
2. 基于设备或 IP 的基础频率限制
3. 上传文件大小与格式限制
4. 单会话并发任务数限制

## 12. 测试范围

### 12.1 功能测试

1. 试衣流程完整可用
2. 试口红流程完整可用
3. 本人上传模式可用
4. 模板模特模式可用
5. 结果页对比展示可用
6. 结果保存可用

### 12.2 异常测试

1. 上传不合格照片时提示准确
2. 供应商失败时任务状态正确
3. 轮询超时时前端提示明确
4. 素材下架后旧前端数据无法继续提交

### 12.3 架构回归测试

1. 更换试衣供应商后，小程序接口无需变更
2. 更换试口红供应商后，小程序接口无需变更
3. 队列或 Worker 重启后，任务状态可恢复或正确失败

## 13. 技术约束与实施原则

1. 小程序端采用 `uni-app`
2. 前端页面结构尽量统一试衣和试色交互，降低学习成本
3. 业务后端必须对 AI 供应商做隔离封装
4. 所有生成任务必须异步执行
5. 首版不为了匿名体验牺牲基本隐私与风控能力

## 14. 后续可扩展方向

本设计刻意为以下能力保留扩展空间：

1. 增加微信登录与历史记录
2. 增加收藏功能
3. 接入内容审核
4. 接入更复杂的风控策略
5. 支持商品跳转或电商化能力
6. 扩展更多美妆试色能力，如腮红、眼影、底妆

## 15. 最终建议

首版按“高真实度效果优先”的原则建设，但不要把高质量 AI 能力直接写死到前端。最合理的落地方式是：

1. `uni-app` 负责交互体验
2. 业务后端负责任务中心和素材系统
3. 两类 AI 能力分别独立适配
4. 运营后台负责持续扩充素材库

这样既能满足当前演示和体验目标，也能在后续正式上线时平滑补齐账号、审核、风控和商业化能力。
